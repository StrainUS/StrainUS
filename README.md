## How I work

I prefer small, verifiable iterations:

- define the product surface before presenting the technology;
- document what a project does and what it does not claim;
- add CI early enough that public repositories can be trusted at a glance;
- use tests where they protect behavior, data invariants, or build reliability;
- keep security notes visible, especially around wallets, keys, public APIs, and experimental contracts.

For blockchain work, my public repositories should be read as prototypes, learning artifacts, or hackathon-grade demonstrations unless explicitly stated otherwise. Smart contracts are not presented as audited production systems.
